## CRITICAL: Stripe API specifics

I've been advised that Stripe has moved to V2 of the Accounts API and 
several patterns have changed. Please follow these specifics — they 
override any older Stripe sample code:

### Use V2 Accounts API for connected accounts

When creating connected accounts:

```js
const account = await stripeClient.v2.core.accounts.create({
  display_name: <supplier name>,
  contact_email: <supplier email>,
  identity: { country: 'gb' },  // UK, not US
  dashboard: 'express',
  defaults: {
    responsibilities: {
      fees_collector: 'application',
      losses_collector: 'application',
    },
  },
  configuration: {
    recipient: {
      capabilities: {
        stripe_balance: {
          stripe_transfers: { requested: true },
        },
      },
    },
  },
});
```

DO NOT pass `type: 'express'` at the top level. That's deprecated V1 
pattern. The V2 equivalent is `dashboard: 'express'` plus a `recipient` 
configuration with capabilities.

### Use V2 Account Links for onboarding

```js
const accountLink = await stripeClient.v2.core.accountLinks.create({
  account: accountId,
  use_case: {
    type: 'account_onboarding',
    account_onboarding: {
      configurations: ['recipient'],
      refresh_url: '<our-app>/admin/suppliers/<id>/onboarding/refresh',
      return_url: '<our-app>/admin/suppliers/<id>/onboarding/complete?accountId=<id>',
    },
  },
});
```

### Use V2 thin webhook events for account updates

V2 accounts use 'thin events' rather than legacy webhook events. Subscribe to:
- v2.core.account[requirements].updated
- v2.core.account[configuration.recipient].capability_status_updated

For local CLI listening:

```bash
stripe listen --thin-events 'v2.core.account[requirements].updated,v2.core.account[configuration.recipient].capability_status_updated' --forward-thin-to <YOUR_LOCAL_ENDPOINT>
```

Parse thin events using `stripeClient.parseThinEvent(req.body, sig, 
webhookSecret)`, then fetch full event data via 
`stripeClient.v2.core.events.retrieve(thinEvent.id)`.

NOT `account.updated`, `capability.updated`, or 
`account.application.deauthorized` (those are V1).

### Get account status via API, not from DB cache

When checking supplier onboarding status:

```js
const account = await stripeClient.v2.core.accounts.retrieve(stripeAccountId, {
  include: ['configuration.recipient', 'requirements'],
});

const readyToReceivePayments = account?.configuration?.recipient
  ?.capabilities?.stripe_balance?.stripe_transfers?.status === 'active';

const requirementsStatus = account.requirements?.summary?.minimum_deadline?.status;
const onboardingComplete = requirementsStatus !== 'currently_due' 
  && requirementsStatus !== 'past_due';
```

Brief in-memory caching (60s) is fine to avoid hammering the API on 
dashboard loads, but always re-fetch when supplier returns from 
onboarding flow.

### Single Stripe Client

Create a single `stripeClient` instance at app initialisation and use it 
throughout. Do not create new instances per request.

```js
const stripeClient = new Stripe(process.env.STRIPE_SECRET_KEY);
```

### CRITICAL: Do NOT use Destination Charges or Stripe Checkout for our model

Stripe's official sample integrations often suggest:
- Destination Charges with `transfer_data.destination` to a single 
  connected account
- Stripe Checkout (hosted payment pages) for "storefront" patterns
- The Products API for pre-defined product catalogues

NONE of these patterns fit our business model. We are NOT a single-merchant 
marketplace. We are a booking agent splitting one client invoice across 
multiple suppliers per transaction.

The correct architecture for us is:

- **Stripe Invoicing** (not Checkout) for client-facing invoices, with 
  itemised line items per supplier
- **Separate Charges and Transfers** (not Destination Charges) — charge 
  the platform account first, then create individual transfers to each 
  supplier's connected account
- Use `source_transaction` parameter on every transfer to link it back to 
  the original charge (this ensures funds are available before the 
  transfer attempts)
- Calculate the platform retention dynamically per gig: 
  (booking commission + reservation fee = Club Kudo Ltd's portion)
- We do NOT need the Products API — there's no product catalogue. Each 
  invoice describes a bespoke gig.

If you encounter Stripe sample code that suggests Destination Charges 
or Checkout, that's because most samples assume single-merchant 
marketplaces. Stick with the architecture above.

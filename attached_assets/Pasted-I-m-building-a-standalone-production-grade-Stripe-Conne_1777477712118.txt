## Core requirements

### Stripe Connect setup
- Use Stripe Connect with Express accounts for suppliers
- Use Separate Charges and Transfers approach (charge platform, transfer to 
  connected accounts after lineup confirmation)
- Implement source_transaction parameter on every transfer to link back to 
  original charge
- All Stripe API calls must use idempotency keys
- All money values stored as integers in pence — never floats
- Webhook signing verification on every incoming Stripe webhook

### Idempotency, reconciliation, and audit trail
- Every Stripe-affecting action logged to an immutable audit table with 
  timestamp, user, action, before/after state, Stripe IDs
- Idempotency keys generated per logical action (not per retry) so 
  retried operations don't double-execute
- Daily reconciliation job that compares your DB state to Stripe's reality 
  via the API and flags any discrepancies for admin review
- Never lose track of money — if something goes wrong, the system flags 
  it for human attention rather than silently failing

### Edge cases that must work from day one
- **Failed transfer**: Charge succeeds but transfer to supplier fails (e.g., 
  account suspended). Surface in admin UI as "needs attention", retry logic 
  with exponential backoff, manual override option.
- **Client refund**: Full and partial refunds. Pull funds back from connected 
  accounts via transfer reversal. Handle the case where supplier has already 
  withdrawn the funds (negative balance handling).
- **Chargeback / dispute**: Funds debited from platform balance, alert admin, 
  freeze any pending transfers for the same gig.
- **Supplier suspension mid-flow**: Account suspended between booking and 
  payout. Flag the gig, allow admin to either reassign supplier or pay 
  manually outside Stripe.
- **Cancelled gig after deposit**: Refund the reservation fee per cancellation 
  policy (full refund 6+ months out, 50% balance retained 3-6 months, full 
  balance retained <3 months). Handle differently if gig had already moved 
  to "lineup confirmed" stage with transfers initiated.
- **Substitution after lineup confirmed**: Original supplier can no longer 
  perform. Replacement supplier takes over. Update gig record, redirect the 
  pending transfer to the new supplier, keep the original substitution 
  documented for audit.
- **Double-booking detection**: Prevent assigning the same supplier to two 
  gigs on the same date.
- **Mid-flight failures**: Charge succeeded but app crashed before transfers 
  initiated. On recovery, detect inconsistent state and reconcile.

### Database schema (proper normalised structure)

Suppliers, clients, gigs, gig_line_items, invoices, transfers, audit_log, 
substitutions, refunds, disputes. Plus reconciliation tables for tracking 
daily Stripe-vs-DB state.

Foreign keys, indexes, soft deletes where appropriate. Timestamps on 
everything (created_at, updated_at). Optimistic locking on records that 
multiple processes might modify.

### Webhooks (full set, not just the basics)
account.updated, account.application.authorized, account.application.deauthorized, 
person.updated, capability.updated, invoice.created, invoice.finalized, 
invoice.paid, invoice.payment_failed, invoice.payment_action_required, 
invoice.voided, charge.succeeded, charge.failed, charge.refunded, 
charge.dispute.created, charge.dispute.closed, transfer.created, 
transfer.failed, transfer.reversed, payout.paid, payout.failed.

Each webhook handler must be idempotent (Stripe retries them) and must 
update the audit log even if the underlying state hasn't changed.

### Admin UI

Built for the agency to operate the business:

1. **Dashboard** — overview of recent gigs, pending payouts, alerts requiring 
   attention, this month's reconciliation status
2. **Gigs** — list with filters (status, date range, supplier, client). 
   Click into a gig to see full timeline: contract, reservation, lineup 
   confirmation, balance invoice, transfers, post-event status
3. **Create/edit gig** — form for new bookings, supplier assignment, line 
   items, pricing
4. **Suppliers** — list, status (onboarding pending / active / suspended), 
   pending payouts, total paid lifetime, button to send onboarding link, 
   link to Stripe Express dashboard
5. **Invoices** — all invoices with status, view/download invoice PDF, 
   resend, void
6. **Transfers** — every transfer with status, retry failed ones, manual 
   override
7. **Reconciliation** — daily reconciliation results, flag investigation queue
8. **Audit log** — searchable, filterable view of every action and event
9. **Refunds & disputes** — workflow for handling, with approval steps
10. **Settings** — VAT registration status, fee structures, webhook config

### Supplier-facing UI

Built so musicians can self-serve and not constantly ask "have I been paid yet":

1. **Login** — magic link via email, no passwords
2. **Dashboard** — upcoming gigs, recent payouts, lifetime earnings
3. **Profile** — their trading name, contact details, instrument(s), bio for 
   client-facing booking pages later
4. **Connect account** — onboarding status, link to Stripe Express dashboard 
   for full payout history
5. **Gigs** — list of past, current, and upcoming bookings with their fee 
   for each gig, payment status
6. **Documents** — invoices issued in their name (under self-billing 
   arrangement), tax statements

### Authentication and authorisation

- Magic-link email auth (Resend or similar) — no passwords
- Role-based access: admin (Gareth, Jake), supplier (musicians, DJs), 
  potentially client-view later
- Suppliers can only see their own data
- Audit who-did-what on every state change

### Infrastructure

- Production Postgres (not just Replit's dev DB — use Neon or similar with 
  backups)
- Background job queue for webhook processing, transfer creation, 
  reconciliation jobs (BullMQ on Redis, or similar)
- Error monitoring (Sentry or equivalent)
- Logging (structured JSON logs, queryable)
- Daily database backups
- Environment variable management for secrets
- Webhook endpoint hosted reliably (not just a Replit dev URL)
- Health check endpoint for monitoring

### Testing

- Unit tests on all money-handling logic
- Integration tests against Stripe's test mode for the major flows: 
  onboarding, invoice, payment, transfer, refund, dispute
- E2E test of the full happy-path flow before considering anything done
